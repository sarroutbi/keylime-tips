Information regarding Capabilities Negotiation:
-----------------------------------------------
The key_class should be asymmetric , key_algorithm rsa, the key_size should match the AK size (probably 2048), the server_identifier should be ak , the local_identifier shopuld be the TPM name (which is a hash over the public portion of the AK, probably we have the calculation implemented somewhere), and the public should be the public portion of the key.
It makes sense to have more than one AK, multiple AK can be generated under the same EK. The EK is unique.

Clarification regarding keys (AKs vs EKs):
------------------------------------------
The properties of an AK (its algorithm, size, usage rules, etc.) are defined in its own creation template when it is generated. The EK parent only serves to protect the new key. Therefore, under a single EK (e.g., an RSA 2048 key), you can create a diverse set of AKs, such as:
* An RSA 2048 AK using the RSASSA signing scheme.
* An RSA 3072 AK using the RSAPSS signing scheme.
* An Elliptic Curve (ECC) AK using the NIST P-256 curve and ECDSA scheme.

Comments regarding AK generation:
---------------------------------
* Ideally, we should only have one created, then, we should persist it to a file, then flush it, to avoid exhausting TPM2 chipset
* Creating more than one AK and send it to the verifier is not "an easy thing". It involves having to register all the keys in the registrar as well.
* For that reason, for push model MVP, we will send an array, but only containing an AK (whose parameters are taken from configuration)

Policy Creation:
----------------
1. Create a runtime policy

Use the keylime-policy tool to generate one. For example:

# From a filesystem scan
keylime-policy create runtime --rootfs / --ramdisk-dir /boot -o policy.json

# From an existing allowlist
keylime-policy create runtime --allowlist allowlist.txt -o policy.json

# From an RPM repository
keylime-policy create runtime --remote-rpm-repo <url> -o policy.json

You can also optionally sign the policy:
keylime-policy sign runtime -r policy.json -k ec-key.pem -o signed-policy.json

2. Upload the policy to the verifier

keylime_tenant -c addruntimepolicy \
  --runtime-policy-name my-policy \
  --runtime-policy policy.json

3. Add an agent referencing that policy

For the push model specifically, you use the --push-model flag and reference the policy by name (no -t target host needed since the agent pushes to the verifier):

keylime_tenant -c add --push-model -u <agent-uuid> \
  --runtime-policy-name my-policy

The same policy can be reused across multiple agents.

Policy management commands

You can also list, show, update, or delete policies:

keylime_tenant -c listruntimepolicy
keylime_tenant -c showruntimepolicy --runtime-policy-name my-policy
keylime_tenant -c updateruntimepolicy --runtime-policy-name my-policy --runtime-policy updated.json
keylime_tenant -c deleteruntimepolicy --runtime-policy-name my-policy

Measured boot policies work the same way (--mb-policy / --mb-policy-name and addmbpolicy, listmbpolicy, etc.).

Documentation pointers

For more detail, check these docs in the keylime repo:
- docs/user_guide/push_model.rst — push model setup
- docs/user_guide/runtime_ima.rst — runtime IMA policy format and details
- docs/user_guide/keylime_policy.rst — the keylime-policy tool usage
- docs/man/keylime_tenant.1.rst — full tenant CLI reference

Adding an agent with a runtime policy:
--------------------------------------

keylime_tenant -c add \
  -t <agent_ip> \
  -u <agent_uuid> \
  --runtime-policy <policy_file.json>

Updating an existing agent's runtime policy:

keylime_tenant -c update \
  -t <agent_ip> \
  -u <agent_uuid> \
  --runtime-policy <new_policy.json>

The runtime policy file is a JSON document that defines which IMA measurements are acceptable. You can generate one using keylime_create_policy:

# Generate a policy from the current system's IMA log
keylime_create_policy -m /sys/kernel/security/ima/ascii_runtime_measurements -o /tmp/min_policy.json
# How to apply it to a running agent
sudo python3 -m keylime.cmd.tenant -c update -u d432fbb3-d2f1-4a97-9ef7-75bd81c00000 --runtime-policy /tmp/min_policy.json --push-mode
sudo python3 -m keylime.cmd.tenant -c update -u d432fbb3-d2f1-4a97-9ef7-75bd81c00000 --runtime-policy /tmp/min_policy.json --runtime-policy-name "production-v1" --push-mode
